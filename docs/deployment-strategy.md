# 本地试运行与服务器迁移策略 V1

> 文档性质：部署与运行环境设计规范，不代表已经开始部署或代码实现  
> 当前版本：0.1  
> 当前阶段：需求与架构设计

## 1. 决策结论

第一版采用以下路线：

```text
Windows 主机
├─ 恢复和验证参考项目
├─ Codex、浏览器、Git与日常开发工具
└─ 必要时承载依赖Windows环境或浏览器登录态的辅助能力

WSL2 Ubuntu
├─ 正式Python项目
├─ SQLite状态库
├─ 采集、标准化、去重、聚类和报告生成
├─ yt-dlp、FFmpeg等通用工具
└─ 本地试运行调度与日志

未来Linux服务器
├─ 迁移WSL2中的正式项目
├─ 复制配置、SQLite与必要原始数据
└─ 将本地调度替换为systemd timer或其他Linux调度方式
```

核心原则是：参考项目可以在原Windows环境中恢复，但正式项目应以Linux兼容方式设计，使WSL2成为本机试运行环境和未来服务器之间的过渡层。

## 2. 为什么采用Windows + WSL2

### 2.1 保留参考项目兼容性

当前参考项目存在Windows绝对路径、本地Python环境和可能尚未找回的`nitter_client.py`等依赖。初期直接在Windows环境恢复，有利于判断原有采集链是否真实可用，而不是先因迁移问题误判其稳定性。

### 2.2 降低未来服务器迁移成本

正式系统在WSL2中使用Linux目录、相对路径和跨平台依赖。未来迁移到Linux服务器时，业务代码、SQLite结构、配置和命令入口应保持不变，主要替换调度与运行环境。

### 2.3 不在早期强制Docker化

第一版仍处于免费Provider验证和需求设计阶段。过早Docker化会额外引入容器网络、Cookie、Windows文件挂载、权限和调试复杂度。

当前要求是保持Docker兼容，而不是立即将所有模块容器化。

## 3. 项目目录原则

正式项目建议放置在WSL2的Linux文件系统：

```text
/home/<user>/projects/news-stugy/
```

不建议长期放置在：

```text
/mnt/c/Users/...
```

原因包括：

- SQLite和大量小文件读写更适合Linux文件系统；
- Git、Python虚拟环境和权限行为更稳定；
- 更接近未来Linux服务器环境；
- 避免Windows与Linux路径规则混用。

所有项目路径必须：

- 使用相对路径或配置化根目录；
- 使用`pathlib.Path`；
- 不写死盘符、用户名或本地缓存路径；
- 允许通过环境变量覆盖数据目录、日志目录和临时目录。

## 4. Windows与WSL2职责边界

### 4.1 Windows负责

- 恢复参考项目原运行环境；
- 寻找和验证`nitter_client.py`及相关依赖；
- 运行Codex、Git客户端、浏览器和编辑工具；
- 必要时承载需要浏览器登录态或Windows专属能力的辅助Provider；
- 使用任务计划程序唤醒电脑或启动WSL2任务。

### 4.2 WSL2负责

- 正式项目主代码；
- SQLite数据库；
- 原始JSON、正文、字幕和日志；
- 内容采集协调；
- 标准化、去重、聚类、分类、评分和报告生成；
- yt-dlp、FFmpeg及音频处理；
- 统一命令入口和本地试运行调度。

### 4.3 禁止的耦合

正式业务代码不得：

- 直接调用固定的Windows盘符；
- 假定某个用户目录存在；
- 将Windows任务计划逻辑写入采集器；
- 将WSL2启动逻辑写入报告生成模块；
- 把浏览器Cookie路径硬编码到通用Provider；
- 依赖人工复制文件完成正常日常运行。

Windows专属能力必须封装为可选Provider或外部命令，并允许未来被服务器端Provider替换。

## 5. 本机运行阶段划分

### 阶段A：手工验证

先手工运行单个模块，不设置正式定时任务：

```text
参考项目恢复
→ 单来源抓取测试
→ 多来源批次测试
→ 标准化和原始归档测试
→ 单次早报/晚报生成测试
```

目标是验证Provider和数据结构，不验证无人值守运行。

### 阶段B：半自动试运行

使用统一命令运行完整流程，但允许人工启动和检查：

```text
collect
→ normalize
→ cluster
→ score
→ build-report
```

要求每一步可单独重跑，失败后不必从头执行。

### 阶段C：本地定时试运行

在采集链和报告结构稳定后，再启用：

- Windows任务计划程序负责唤醒或启动WSL2；
- WSL2内部负责执行统一命令；
- 早报、晚报和周报按北京时间运行；
- 电脑未开机或任务错过时，支持启动后的补偿运行。

### 阶段D：服务器迁移

连续本地试运行两个完整交易周后，评估迁移Linux服务器。

迁移重点不是重写代码，而是：

- 安装相同依赖；
- 复制配置和密钥；
- 复制SQLite和必要原始数据；
- 将任务计划替换为Linux调度器；
- 重新测试免费Provider在服务器IP环境下的成功率。

## 6. 调度原则

调度器只负责调用任务，不包含业务规则。

统一命令应类似：

```text
news-study collect --until <time>
news-study process --window <start> <end>
news-study report morning --date <date>
news-study report evening --date <date>
news-study report weekly --date <date>
```

具体命令形式可在实现阶段调整，但必须满足：

- 可以手工执行；
- 可以重复执行；
- 可以指定时间窗口；
- 可以从失败步骤继续；
- 可以在Windows任务计划、cron和systemd timer中使用同一入口。

## 7. 电脑休眠与错过任务

本机环境不能假定08:00和18:00一定在线，因此必须设计补偿运行。

建议规则：

1. 每次系统启动或WSL2任务启动时检查最近应完成的报告；
2. 如果计划任务已错过但仍在允许补偿时间内，则补生成；
3. 补生成仍使用原报告窗口，不把启动后的新内容混入旧报告；
4. 同一报告只能有一个正式版本，重复运行时更新或跳过，而不是生成多个副本；
5. 补生成报告应记录实际生成时间和计划生成时间。

建议补偿范围：

- 早报：当日中午前可以补生成；
- 晚报：当日结束前可以补生成；
- 周报：次日内可以补生成；
- 超过补偿范围后默认只回填数据库，不再补发过期日报，具体规则后续再确认。

## 8. 幂等与断点恢复

所有自动任务必须按幂等方式设计：

- 同一来源、外部ID和内容哈希不得重复写入Item；
- 同一报告类型、计划时间和窗口只能对应一个Report；
- 采集成功、部分成功和失败状态必须持久化；
- AI处理结果应保存版本和输入引用，避免重复消耗；
- 单个Provider失败后可以单独重试；
- 报告生成失败不应删除此前完成的采集和聚类结果。

建议任务状态：

```text
pending
running
partial_success
success
failed_retryable
failed_final
skipped_already_done
```

## 9. 数据与备份

本地阶段至少备份：

- SQLite数据库；
- 配置文件；
- 报告成品；
- Provider游标和关键运行状态；
- 无法重新获取的重要原始正文或字幕。

不要求永久备份所有临时音频和中间文件。

建议：

- SQLite每日生成一次可恢复备份；
- 报告与配置进入Git或独立备份；
- 密钥不进入Git；
- 原始数据按保留策略定期清理；
- 服务器迁移前执行一次完整一致性检查。

## 10. 未来服务器方案

未来优先使用普通Linux服务器，而不是为迁移重新设计整个系统。

服务器端建议：

- Python虚拟环境或容器；
- systemd service + systemd timer；
- SQLite继续作为V1数据库；
- 定期备份到独立磁盘或对象存储；
- 日志轮转；
- Provider健康检查；
- 必要时保留本地Windows采集节点。

如果X或公众号在服务器IP下明显劣化，可采用混合模式：

```text
Windows本地采集不稳定平台
→ 输出RawItem或原始JSON
→ 同步到服务器
→ 服务器完成后续处理和报告生成
```

混合模式只是备用，不作为第一版默认架构。

## 11. 当前阶段暂不实施

在继续完成需求和架构设计期间，暂不：

- 安装WSL2依赖；
- 创建正式Python工程；
- 配置Windows任务计划；
- 编写systemd timer；
- 部署Docker；
- 搬迁SQLite；
- 购买或开通服务器。

当前只确定部署边界、迁移原则、调度要求和故障恢复规则。

## 12. 后续待确认事项

- 本地电脑通常的开机和休眠时间；
- 错过早报、晚报后的补偿时限；
- 原始正文、字幕、音频和图片的保留周期；
- SQLite备份位置；
- AI任务失败时的最大重试次数；
- 未来服务器所在地区及对X、公众号、YouTube的网络影响；
- 是否需要保留Windows本地采集节点。
