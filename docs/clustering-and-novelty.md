# 信息聚类、增量判断与跨报告更新规则 V1

> 文档性质：智能处理层设计规范，不代表已经开始代码实现  
> 当前版本：0.1  
> 当前阶段：需求与架构设计

## 1. 设计目标

系统不能把相似内容简单删除，也不能把讨论同一大主题的所有内容强行合并。聚类层需要解决四个问题：

1. 哪些Item描述的是同一个现实事件；
2. 哪些观点是在回答同一个命题；
3. 新内容相对已有信息增加了什么；
4. 早报已经出现的主题，晚报或周报是否值得再次展示。

总体链路：

```text
Item
→ 候选Cluster检索
→ 关系判定
→ 合并、拆分或新建Cluster
→ 提取novel_information
→ 更新Cluster与Event
→ 判定跨报告update_type
→ 进入评分与报告选择
```

核心原则：

- 先确定“是否同一对象”，再判断文本相似度；
- 文本相似不等于同一事件，文本不同也可能是同一事件的新增进展；
- 错误合并通常比暂时重复更危险；判断不确定时优先保留独立Cluster；
- 报告展示Cluster，不直接堆叠Item；
- 同一事件的事实、市场反应、产业影响和观点可以关联，但不得互相替代；
- 相反观点并列保留，不取平均值；
- 所有关系判定、增量字段和版本都必须可追溯。

## 2. 三类Cluster边界

### 2.1 Event Cluster：事件簇

Event Cluster描述一个可识别的现实事件、数据发布、政策动作、公司动作或持续进程中的一次明确状态变化。

事件身份至少由以下字段组合决定：

```text
主体 actor
+ 动作 action
+ 对象 object
+ 时间或阶段 time/state
+ 地区或市场 jurisdiction/market
```

例如：

- “美联储7月会议维持利率不变”是一个事件；
- “主席会后表示9月仍取决于数据”属于同一会议事件的新增细节或政策指引；
- “另一位官员次日支持降息”是新的发言事件，但可以关联到同一个持续Event；
- “美国6月CPI公布”与“美国7月CPI公布”是两个不同事件，不能因名称相似合并。

Event Cluster适合：

- 宏观数据发布；
- 央行决议和官员讲话；
- 政策、监管、关税和法律进展；
- 公司财报、指引、融资和重大交易；
- 矿山停产、制裁、出口限制和供应扰动；
- ETF审批、交易所事件、黑客攻击和脱锚；
- 重要市场异动及其后续确认。

### 2.2 Opinion Cluster：观点簇

Opinion Cluster描述多个作者针对同一个可表达命题的判断，而不是泛泛的共同主题。

一个观点命题应尽量结构化为：

```text
对象 subject
+ 方向或结论 stance
+ 时间周期 horizon
+ 核心依据 rationale
+ 催化剂 catalyst
+ 失效条件 invalidation
```

例如：

- “美国流动性将在三季度改善并利好风险资产”可以形成一个观点簇；
- “铜长期供给短缺”与“本周铜价会反弹”不是同一观点命题；
- 两位作者都讨论黄金，但一个谈央行购金、一个谈技术形态，不应只因标的相同就合并；
- 对同一命题看多与看空的观点放在同一观点簇中，以`contradiction`并列。

### 2.3 Research Cluster：研究簇

Research Cluster用于方法、框架、专题报告、深度视频和教育内容。

默认规则：

- 一篇独立研究或一支深度视频先形成独立研究簇；
- 同一作者在X发布摘要、YouTube发布完整视频、公众号发布全文时，可关联为同一研究簇；
- 多篇研究只有在研究问题、方法和结论高度一致时才合并；
- 研究簇更关注框架、证据和可复用方法，不因当日市场未波动而自动降级；
- 日报通常只选最相关结论，完整研究更适合周报或研究栏目。

## 3. Cluster与持续Event的关系

Cluster是报告处理单位，Event是跨天、跨周跟踪的持续状态对象。

```text
持续Event：美国精炼铜关税政策
├─ Cluster A：政府启动调查
├─ Cluster B：官员提交政策建议
├─ Cluster C：总统宣布关税安排
├─ Cluster D：实施细则发布
└─ Cluster E：政策生效后的市场反应
```

因此：

- 一次明确进展形成一个Event Cluster；
- 多个Event Cluster可以关联同一个持续Event；
- Event保存生命周期和当前阶段；
- Cluster保存本次信息、来源、观点和市场反应；
- 周报按Event演变重组，不简单复制一周内的Cluster摘要。

## 4. 候选Cluster检索

V1不建立独立向量数据库，也不允许仅凭Embedding相似度自动合并。

候选检索顺序：

```text
1. 平台ID、标准化URL、正文哈希等确定性键
2. 同一持续Event或官方事件ID
3. 实体、标的、地区、动作和事件阶段过滤
4. 时间窗口过滤
5. 标题、关键词和SQLite FTS相似候选
6. 对少量候选执行结构化AI关系判定
```

### 4.1 默认时间窗口

这些窗口只用于寻找候选，不代表超过窗口一定不能合并：

| Cluster类型 | 默认候选窗口 | 说明 |
|---|---:|---|
| 突发事件与新闻 | 前后72小时 | 避免跨事件错误合并 |
| 宏观数据发布 | 同一统计期与发布批次 | 不按文本相似度跨月份合并 |
| 央行会议、财报、拍卖 | 同一会议、财季或批次 | 使用官方日期和事件ID优先 |
| 政策与监管进程 | 30天 | 通过持续Event可扩展到更长 |
| 观点簇 | 14天 | 长周期观点可扩展到90天 |
| 研究簇 | 90天 | 同一研究系列可更长 |
| 周报主题重组 | 7天主窗口，必要时回看30天 | 用于判断主题延续与拐点 |

### 4.2 候选数量

- 每个新Item最多送入20个候选Cluster进行精判；
- 先按事件身份和时间缩小范围，再考虑文本相似；
- 找不到可信候选时直接新建Cluster；
- 候选过多说明分类字段过宽，应收紧实体、动作或时间条件，而不是让模型盲选。

### 4.3 Embedding使用边界

V1允许将Embedding作为可选辅助特征，并缓存在SQLite或ProcessingArtifact中，但：

- 不引入独立向量数据库；
- 不以固定余弦阈值直接合并；
- Embedding只用于候选排序或发现漏掉的相近内容；
- 最终关系仍需依据事件身份、事实字段和结构化判定。

## 5. Item与Cluster关系

每个Item加入Cluster时至少指定一个主关系：

| relation | 含义 | 是否构成增量 |
|---|---|---|
| `duplicate` | 同稿、转载、改写或没有实质新增 | 否 |
| `corroboration` | 独立来源确认同一核心事实 | 增加确认度，不一定增加事实 |
| `new_detail` | 增加数字、时间、范围、主体、机制或执行细节 | 视重要性而定 |
| `update` | 事件阶段或状态发生推进 | 通常是 |
| `market_reaction` | 新增价格、利率、成交、资金流或跨资产反应 | 视持续性而定 |
| `industry_impact` | 新增产业链、公司、供需或盈利影响 | 视重要性而定 |
| `commentary` | 对事实进行解释、归因或推演 | 观点增量 |
| `contradiction` | 对核心事实、因果或前景提出相反判断 | 是，必须保留 |
| `correction` | 修正先前错误数字、时间、主体或口径 | 是 |
| `retraction` | 原来源撤回、否认或宣布此前信息无效 | 高度重要 |

一个Item可以带辅助标签，但只保存一个主关系，避免关系字段失控。

### 5.1 `duplicate`与`corroboration`的区别

`duplicate`：

- 相同通讯稿、转载、编译或复述；
- 信息来源最终都指向同一个原始出处；
- 没有独立核实过程；
- 不增加独立来源数量。

`corroboration`：

- 来源之间具备独立性；
- 独立获得或核实同一事实；
- 可以提高事实置信度；
- 增加`independent_source_count`。

媒体数量多不自动等于独立确认。多家媒体引用同一匿名信源或同一公告，仍可能只算一个原始来源。

### 5.2 `new_detail`的判定

以下通常属于新增细节：

- 新的数字、比例、金额、数量或区间；
- 明确的日期、期限、生效时间或会议安排；
- 新增受影响主体、地区、产品或资产；
- 新增政策豁免、条件、执行机制或法律依据；
- 新增供应量、库存、产能、资本开支或盈利影响；
- 新增观点依据、催化剂或失效条件。

但新增一个无关紧要的措辞、重复背景或轻微改写不算`new_detail`。

## 6. 是否属于同一Cluster的判定

### 6.1 Event Cluster合并条件

原则上同时满足：

1. 核心主体一致或存在明确上下级关系；
2. 核心动作与对象一致；
3. 时间、统计期、会议批次或事件阶段兼容；
4. 新内容可以被解释为同一事件的确认、细节、进展、影响或反应；
5. 合并后不会让标题、事实结论或时间线产生歧义。

以下情况应拆分：

- 同一主体在不同日期采取不同政策动作；
- 同名宏观数据属于不同月份或不同修订批次；
- 同一公司发布财报与随后单独宣布并购；
- 同一商品发生两个独立供应中断；
- 一个是传闻，一个是与传闻无直接关系的其他事实；
- 合并后无法准确表达谁在何时做了什么。

### 6.2 Opinion Cluster合并条件

原则上需要：

- 讨论同一个可明确表达的命题；
- 时间周期大致一致；
- 结论可以映射为支持、反对或条件性判断；
- 依据差异可以作为观点分歧保留。

只讨论同一个资产、行业或宏观主题不足以合并。

### 6.3 Research Cluster合并条件

需要至少满足两项：

- 同一原创研究在不同平台的版本；
- 同一研究问题和方法；
- 同一系列的连续更新；
- 新版本主要是扩展数据或案例，而非完全不同的研究。

### 6.4 保守拆分原则

出现以下任一情况时，默认不自动合并：

- 事件主体或时间不明确；
- 只有标题或残缺摘要；
- 两个候选都可能成立；
- 合并会改变事实置信度；
- 需要依赖模型猜测隐藏上下文；
- 相似度高，但核心动作或统计期不同。

后续获得更多信息时允许再合并Cluster；错误合并必须支持拆分和关系重算。

## 7. Cluster内部结构

Event Cluster建议维护：

```yaml
fact_base: 已确认的核心事实
current_state: 当前阶段或状态
new_details: 新增数字、范围、时间表和机制
market_reactions: 市场与跨资产反应
industry_impacts: 产业链和公司影响
interpretations: 来源观点与解释
contradictions: 事实或观点冲突
unknowns: 尚未确认的问题
representative_sources: 代表性来源
```

Opinion Cluster建议维护：

```yaml
proposition: 核心命题
horizon: 时间周期
supporting_views: 支持观点
opposing_views: 反对观点
shared_evidence: 共用证据
key_disagreements: 分歧所在
catalysts: 催化剂
invalidation_conditions: 失效条件
```

Research Cluster建议维护：

```yaml
research_question: 研究问题
method_or_framework: 方法或框架
key_findings: 核心发现
supporting_evidence: 证据
limitations: 局限
practical_relevance: 对当前市场或研究工作的意义
```

Cluster摘要每次更新时不得只覆盖旧摘要，应保留版本与新增信息来源。

## 8. 信息增量与跨报告update_type

Item关系用于描述Item对Cluster的贡献；`update_type`用于判断同一Cluster是否值得再次进入后续报告。

第一版继续使用：

```text
no_change
minor_detail
material_update
reversal
market_confirmation
```

### 8.1 `no_change`

- 只有重复转载、换标题或背景复述；
- 新来源没有增加独立确认；
- 新观点与此前表达实质相同；
- 不改变Cluster任何核心字段。

默认不重复报告。

### 8.2 `minor_detail`

- 新增非关键背景、措辞或小范围数字；
- 不改变事件阶段、影响范围、结论、置信度或紧迫性；
- 即使删除该细节，报告核心结论仍完全相同。

默认不重复报告，可更新Cluster内部记录。

### 8.3 `material_update`

满足以下任一原则即可考虑：

- 官方确认、批准、实施、否认或进入新生命周期阶段；
- 关键金额、比例、数量、时间表或影响范围发生明显变化；
- 新信息改变核心标题或第一句话；
- 新信息改变受影响资产、行业或跨资产传导判断；
- 新信息显著提高或降低事实置信度；
- 新信息改变事件紧迫性、持续时间或后续观察重点；
- 出现重要产业影响、公司影响或政策执行细节；
- 重要反方证据使原判断需要重新表述。

操作性测试：

> 如果加入新信息后，报告的核心结论、影响判断、置信度、关注对象或后续观察至少有一项必须改变，则属于material_update。

### 8.4 `reversal`

- 官方撤回、否认或反转此前决定；
- 实际结果与此前已确认方向相反；
- 核心数据被大幅修订；
- 原有市场叙事被可靠证据证伪；
- 观点作者明确改变方向且提供新依据。

`reversal`必须提高优先级，并明确说明“此前是什么、现在改变了什么”。

### 8.5 `market_confirmation`

不是出现任何价格波动都算确认。至少满足其中一项：

- 事件发生后，相关核心资产出现方向一致且可识别的持续反应；
- 反应从单一资产扩展到利率、汇率、股票、商品或加密等多个相关市场；
- 成交量、波动率、资金流或期限结构支持该反应；
- 此前影响不明确，后续市场定价使影响方向更清晰；
- 高质量来源将该事件列为主要驱动之一，且时间关系与市场表现吻合。

无法排除多个同时发生的驱动时，应标记为`likely`或`unclear`，不能强行写成确认因果。

## 9. 早报、晚报和周报的重复规则

### 9.1 早报到晚报

早报已经展示的Cluster，晚报只有在出现以下情况时再次进入核心部分：

- `material_update`；
- `reversal`；
- `market_confirmation`；
- 重大强制事件出现新的官方状态；
- 上午信息此前只有线索，日间获得可靠确认并改变事实置信度。

仅有`corroboration`时：

- 如果此前事实置信度较低，而独立确认使其达到可报告标准，可升级为`material_update`；
- 如果此前已经充分确认，则只更新来源数量，不重复展示。

晚报重复出现时必须突出“新增部分”，不能复制早报原文。

### 9.2 跨日重复

持续事件可以跨日出现，但必须满足：

- 生命周期阶段变化；
- 新增关键数据、实施细节或影响范围；
- 市场出现新的显著定价；
- 原判断被修正或反转；
- 事件仍为当日最重要驱动，且报告明确说明今日新增内容。

单纯“事件仍在持续”不足以重复报道。

### 9.3 周报

周报不受日报“必须有单次material_update”限制。某主题即使每天都只有小更新，只要在一周内表现出持续性、累积效应或结构性意义，也可以进入周报。

周报处理方式：

```text
本周相关Cluster
→ 按持续Event或核心命题重新聚合
→ 构建时间线
→ 区分起点、关键进展、市场反应和当前状态
→ 提炼共识、分歧和下周观察点
```

周报不得拼接日报摘要，应重新写出：

- 本周发生了什么变化；
- 哪些信息只是重复确认；
- 市场如何逐步定价；
- 哪些观点被验证或证伪；
- 下周最关键的状态变量是什么。

## 10. 跨平台与同一作者去重

同一创作者在多个平台发布同一主题时：

- X预告 + YouTube完整视频：以完整视频或文章为主，X作为发现或摘要关系；
- YouTube视频 + 公众号全文：选择内容更完整、可引用性更好的版本作为canonical item；
- 同一作者跨平台逐步增加信息：合并到同一研究簇或观点簇，新增部分记为`new_detail`；
- 同一作者重复表达相同观点，不增加独立来源数量；
- 不同团队成员若信息来源和研究过程独立，可根据实际情况计算独立来源。

一份报告中同一作者原则上只保留一个核心主题，除非涉及两个相互独立的重大事件。

## 11. Canonical Item与代表性来源

### 11.1 Event Cluster

优先顺序：

```text
官方原始文件或公告
→ 直接当事方或公司公告
→ 最完整的高质量独立报道
→ 可访问且内容完整的专业分析
→ 社交媒体线索
```

### 11.2 Opinion与Research Cluster

优先顺序：

```text
作者完整原文、视频或研究报告
→ 作者跨平台完整版本
→ 高质量摘要或访谈
→ 二次转载与摘录
```

每个Cluster报告中通常展示1至3个代表性来源：

- 一个核心原始来源；
- 一个独立确认或高质量背景来源；
- 一个重要分歧或产业影响来源。

不因来源很多而罗列大量重复链接。

## 12. 自动判定输出结构

每次Item与候选Cluster的关系判定应输出结构化结果：

```yaml
decision: merge | new_cluster | uncertain
cluster_id: string | null
cluster_type: event | opinion | research
```

## 13. Cluster拆分、合并与修复

系统必须允许后续修复：

### 合并

当后续证据证明两个Cluster是同一事件时：

- 选择保留Cluster；
- 迁移Item关系；
- 重算来源数量、摘要、增量和报告引用；
- 保存合并历史，不删除旧ID映射。

### 拆分

当发现错误合并时：

- 创建新的Cluster；
- 按事实身份重新分配Item；
- 重算两个Cluster；
- 标记受影响的历史报告，但不静默改写已发布报告；
- 在后续报告中必要时进行更正。

### 关闭

- 一次性事件完成且短期不再更新：`closed`；
- 持续Event下的旧Cluster可以关闭，但Event保持活跃；
- 长期无更新且仅供历史查询：`archived`。

## 14. 第一版验收指标

试运行时至少记录：

- 精确重复压缩率；
- 近似重复合并率；
- 抽样错误合并率；
- 抽样漏合并率；
- `duplicate`与`corroboration`混淆案例；
- 早报到晚报无意义重复率；
- `material_update`判定正确率；
- Cluster平均Item数量和独立来源数量；
- 发生拆分或更正的Cluster数量；
- 周报主题是否真正体现演变，而非日报拼接。

第一版优先控制错误合并和无意义重复，不追求把所有相似内容都自动归入同一Cluster。
