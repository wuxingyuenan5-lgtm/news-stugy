# 内容处理与智能分析流水线 V1

> 文档性质：处理层设计规范，不代表已经开始代码实现  
> 当前版本：0.1  
> 当前阶段：需求与架构设计

## 1. 设计目标

采集到内容不等于可以直接交给模型写报告。第一版需要建立一条可追溯、可重跑、可失败隔离的处理流水线：

```text
发现内容
→ 获取正文、字幕、图片或数据
→ 标准化
→ 确定性过滤与精确去重
→ 内容级结构化分析
→ 相似候选检索
→ 事件簇、观点簇和研究簇更新
→ 信息增量判断
→ 多维评分与报告候选分层
→ 报告综合
→ 事实与引用检查
→ Markdown输出
```

核心原则：

- 先使用确定性规则，再使用AI；
- 先去除完全重复，再进行昂贵分析；
- 报告处理Cluster，不直接堆叠Item；
- 事实、解释、观点、预测和交易判断分开；
- 内容不完整时限制可作出的结论；
- 每一步保存输入、输出、版本和状态；
- 同一输入未变化时优先复用已有ProcessingArtifact；
- 单个Item、模型调用或Provider失败不得阻塞整份报告。

---

## 2. 总体阶段

```text
Stage 0  Discovery                 发现新内容
Stage 1  Acquisition               获取完整内容
Stage 2  Normalization             标准化
Stage 3  Deterministic Filtering   确定性过滤
Stage 4  Exact Deduplication       精确去重
Stage 5  Item Analysis             单条内容结构化分析
Stage 6  Candidate Retrieval       相似候选检索
Stage 7  Cluster Update            信息簇更新
Stage 8  Event Update              持续事件状态更新
Stage 9  Novelty Classification    增量类型判断
Stage 10 Scoring                   多维评分
Stage 11 Report Selection          报告候选选择
Stage 12 Report Synthesis          报告综合
Stage 13 Quality Control           事实、引用和完整性检查
Stage 14 Publication               Markdown输出与状态更新
```

各阶段必须能够单独重跑，不能要求每次失败后从采集开始全部重做。

---

## 3. Stage 0：内容发现

Discovery只负责找到可能的新内容，并保存最小元数据：

- 来源；
- 平台外部ID；
- 标题或帖子文本；
- 原始链接；
- 原始发布时间；
- 发现时间；
- 使用的Provider；
- 引用、转发、线程或视频状态等平台字段。

发现阶段不负责：

- 判断内容是否重要；
- 生成最终摘要；
- 判断与其他来源是否属于同一事件；
- 根据标题推断不可访问正文。

只取得标题、时间和链接时，Item标记为`metadata_only`。

---

## 4. Stage 1：内容获取

### 4.1 内容完整度

统一使用：

| 状态 | 含义 | 可用于详细摘要 |
|---|---|---|
| `complete` | 完整正文、完整帖子线程、有效字幕或完整数据 | 是 |
| `partial` | 只有部分正文、摘要、部分线程或不完整字幕 | 有限制 |
| `metadata_only` | 只有标题、时间和链接 | 否，只能用于线索发现 |
| `unavailable` | 本次获取失败 | 否 |

内容完整度不等于来源质量，也不等于事件重要性。它只限制系统能够可靠陈述什么。

### 4.2 X

获取后应拆分并保存：

- 原创文本；
- 引用帖原文；
- 作者新增评论；
- 转发对象；
- 自回复线程顺序；
- 回复对象；
- 帖子ID和原始链接；
- 精确或近似时间。

纯转发可以作为传播和发现信号，但没有新增评论时不单独形成观点。

### 4.3 微信公众号

保存：

- 标题、作者、公众号与真实发布时间；
- 清洗前HTML；
- 清洗后正文；
- 图片及其在正文中的位置；
- 长图或图片型正文的视觉理解结果；
- 文章是否只能取得部分内容；
- 原始链接和种子公众号身份。

付费、广告和软文可以过滤，但知识科普、量化方法和图片文章不能因格式特殊而直接排除。

### 4.4 YouTube

内容获取顺序：

```text
人工字幕
→ 自动字幕
→ 音频转写
→ 必要时关键画面理解
```

排除直播、直播回放和Shorts。无法取得字幕、音频和有效画面时，不根据标题生成内容摘要。

### 4.5 专业媒体和网页

优先获取公开正文。付费墙内容只使用合法可见的标题、导语、摘要和元数据，不绕过付费限制，也不推断不可见内容。

---

## 5. Stage 2：标准化

不同平台统一形成Item，至少完成：

- URL标准化；
- 时间统一为UTC存储，并保留原始时间；
- 语言识别；
- HTML和格式噪音清理；
- 空白、重复页眉页脚和追踪参数清理；
- X线程按顺序合并；
- 字幕按时间戳整理；
- 公众号图片和正文顺序保留；
- 正文哈希与标准化哈希生成；
- 原始文件路径记录。

标准化不能改变原意。原始内容必须保留，后续错误可以回溯。

---

## 6. Stage 3：确定性过滤

在调用AI前先排除：

- 发布时间不属于处理窗口且不属于补偿回填；
- X对其他用户的普通回复；
- YouTube直播、直播回放和Shorts；
- 完全重复的外部ID、URL或正文哈希；
- 纯转发且没有新增评论；
- 明确的抽奖、招生、纯广告和生活内容；
- 无正文、无字幕且无可用元数据；
- 与覆盖市场和研究主题完全无关；
- 已经处理成功且输入未变化。

需要语义判断的“是否有研究价值”“是否只是事后归因”不应伪装成确定性规则，应进入轻量分类阶段。

---

## 7. Stage 4：精确去重

精确去重顺序：

```text
平台 + 来源 + 外部ID
→ 标准化URL
→ 原始正文哈希
→ 来源 + 发布时间 + 标准化标题
→ 标准化内容哈希
```

精确重复记录不进入后续AI分析，但原始ProviderAttempt仍保留，便于统计不同Provider的覆盖和重复情况。

---

## 8. Stage 5：单条内容结构化分析

### 8.1 输出结构

每条可用Item生成结构化ProcessingArtifact：

```yaml
language: zh | en | other
content_nature:
  - fact
  - data
  - analysis
  - opinion
  - forecast
  - trading_view
  - education
summary: string
facts:
  - text: string
    evidence_span: string | null
    confidence: high | medium | low
opinions:
  - text: string
    author_attribution: string
forecasts:
  - text: string
    horizon: intraday | days | weeks | months | years | unknown
    conditions: string | null
data_points:
  - name: string
    value: string
    period: string | null
    source_context: string | null
entities: [string]
markets: [string]
topics: [string]
time_horizon: string | null
promotional_ratio: low | medium | high
quality_flags: [string]
```

### 8.2 事实与观点

模型必须明确区分：

- 已发生的事实；
- 引用的外部事实；
- 作者解释；
- 作者预测；
- 交易观点；
- 教学方法。

不能把“作者认为”改写成已经确认的事实。

### 8.3 证据范围

重要事实尽可能保存支持它的原文片段、字幕时间戳或数据字段。报告生成时优先使用有证据范围的事实。

---

## 9. 长内容处理

### 9.1 YouTube和长篇文章

不把数小时视频或数万字文章一次性直接交给报告模型。

```text
按章节、时间戳或语义边界切片
→ 分片提取事实、观点和数据
→ 删除片内重复
→ 合并主题
→ 生成内容级综合结果
```

建议分片目标：

- 每片内容长度适合单次可靠分析；
- 视频保留开始与结束时间戳；
- 文章保留段落或标题位置；
- 分片之间保留少量重叠，避免观点在边界被截断；
- 最终摘要必须基于分片结果，而不是仅基于标题与描述。

### 9.2 图片与长图

图片分析仅在图片承载研究信息时启动，例如：

- 数据图表；
- 研究结论长图；
- 公众号图片型正文；
- 视频演示中的关键图表。

装饰图、头像、广告图和重复封面不进入视觉分析。

---

## 10. Stage 6：相似候选检索

语义聚类不能对全部历史Item进行全量两两比较。

候选范围先通过确定性条件缩小：

- 时间距离；
- 主市场与主题；
- 关键实体；
- 事件类别；
- 标题和关键词；
- 内容性质。

然后再使用语义相似度或AI判断关系。

第一版不强制使用向量数据库。可以先使用：

```text
SQLite候选过滤
+ 文本相似度
+ 必要的模型关系判断
```

后续数据规模明显增大时再评估向量索引。

---

## 11. Stage 7：Cluster更新

### 11.1 Cluster类型

- `event`：同一现实事件、数据发布、政策或公司行为；
- `opinion`：围绕同一命题或资产判断的观点；
- `research`：方法、框架、长篇研究和教学内容。

### 11.2 Item与Cluster关系

- `duplicate`：实质相同；
- `corroboration`：独立确认；
- `new_detail`：新增数字、范围或细节；
- `update`：事件阶段变化；
- `market_reaction`：新增市场反应；
- `industry_impact`：新增产业影响；
- `commentary`：解释与观点；
- `contradiction`：相反事实判断或相反观点。

相反观点不得平均化成模糊结论，应在同一主题下并列展示。

### 11.3 独立来源

同时记录：

- `source_count`：总来源数量；
- `independent_source_count`：独立来源数量。

大量转载不应机械提高重要性。

---

## 12. Stage 8：Event更新

Cluster与持续Event关联后，判断状态是否变化：

```text
rumor
→ proposal
→ consultation
→ review
→ approved
→ enacted
→ effective
→ implementation
→ enforcement
```

同时允许`reversed`。

只有状态变化、执行细则、新范围、新时间表或实质市场影响才构成值得重复报告的更新。

---

## 13. Stage 9：信息增量判断

与现有Cluster和此前报告比较，输出：

- `no_change`：没有新增；
- `minor_detail`：轻微细节；
- `material_update`：实质变化；
- `reversal`：方向性反转；
- `market_confirmation`：市场反应显著验证或证伪此前判断。

建议同时提取增量类型：

```yaml
new_facts: [string]
new_numbers: [string]
new_scope: [string]
new_timeline: [string]
new_official_confirmation: [string]
new_market_reaction: [string]
new_interpretation: [string]
```

早报已报道的事件，晚报只有`material_update`、`reversal`或`market_confirmation`才再次进入正文。

---

## 14. Stage 10：评分

评分对象是Cluster，不是单条Item。

日报权重：

- 市场影响；
- 信息增量；
- 来源质量；
- 时效性；
- 跨资产关联；
- 用户关注匹配；
- 证据质量。

周报权重：

- 结构性重要性；
- 主题持续性；
- 一周演变；
- 跨资产关联；
- 证据质量；
- 用户关注匹配；
- 新颖性。

总分用于候选分层，不构成机械入选规则。重大官方事件、系统性风险和方向性反转可以覆盖普通分数门槛。

---

## 15. Stage 11：报告候选选择

候选分为：

- `core`：核心结论和核心事件；
- `attention`：重点观点、分市场补充；
- `browse`：其他值得浏览；
- `archive`：只归档。

选择规则：

- 不按平台和市场凑数；
- 同一事件只出现一次，跨资产影响写在同一主题中；
- 同一作者原则上一份报告只保留一个核心主题；
- 教育与方法内容优先进入周报；
- 交易观点使用更高门槛；
- `metadata_only`不能生成详细内容；
- 内容不足时宁可缩短报告。

---

## 16. Stage 12：报告综合

报告模型读取的是：

- 已选择Cluster；
- 每个Cluster的事实、观点、增量和来源；
- 市场快照；
- 已公布宏观数据；
- 后续事件日历；
- 来源完整度和运行异常；
- 此前报告引用状态。

单个核心主题建议包含：

```markdown
### 标题

**事实：** 已确认发生了什么。

**影响：** 对哪些资产、预期或产业链有何影响。

**判断与不确定性：** 当前解释、分歧和尚未确认事项。

**来源：** 原始来源与重要独立确认。
```

不明确的因果关系应写“可能驱动之一”“影响暂不明确”或“目前没有单一清晰解释”。

---

## 17. Stage 13：质量控制

最终输出前自动检查：

- 每个重要事实是否有来源；
- 是否将观点写成事实；
- 是否根据标题推断不可见正文；
- 是否遗漏事件状态或时间；
- 是否重复早报或此前日报；
- 是否把转载数量当作独立确认；
- 是否引用了`metadata_only`内容生成详细摘要；
- 数据单位、方向和债券BP变化是否正确；
- 报告时间窗口是否正确；
- 标题、链接和发布时间是否可回溯；
- 是否为填满栏目加入低价值内容。

检查失败分为：

- 可自动修复；
- 重新生成局部段落；
- 删除不可靠内容；
- 标记报告为`partial`但按时输出；
- 阻止发布，仅限报告主体无法可靠生成的严重情况。

## 18. 模型路由原则

第一版不绑定具体AI供应商，但区分任务等级：

### 轻量任务

- 语言识别；
- 初步相关性；
- 内容性质分类；
- 简单实体和主题提取；
- 候选相似关系筛选。

可以使用速度更快、成本更低的模型。

### 高质量任务

- 长内容综合；
- 复杂事实与观点拆分；
- Cluster合并与增量判断；
- 相反观点处理；
- 重大事件影响分析；
- 最终报告综合和质量修复。

应使用更可靠的强模型。

### 视觉任务

仅在图片或画面承载关键研究信息时调用视觉模型。

### 转写任务

无字幕视频使用音频转写模型。转写结果仍需经过内容分析，不能直接作为报告摘要。

所有模型调用记录：

- 模型供应商；
- 模型名称；
- 输入哈希；
- Prompt版本；
- 输出状态；
- 处理时间；
- 必要时的Token和成本。

---

## 19. 状态流转

Item建议状态：

```text
discovered
→ fetched
→ normalized
→ filtered 或 analyzed
→ clustered
→ scored
→ selected
→ reported
→ archived
```

任一步可以进入：

```text
failed_retryable
failed_final
```

失败状态不覆盖此前成功产物。输入修复后可从失败阶段继续。

---

## 20. 当前不做

第一版暂不建设：

- 全量知识图谱；
- 自动因果网络；
- 复杂向量数据库；
- 博主历史全量画像；
- 自动交易信号和仓位建议；
- 实时逐笔新闻终端。

第一版重点是把“内容获取—去重—聚类—增量—评分—报告”闭环稳定跑通。
