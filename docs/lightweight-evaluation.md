# 轻量评测与验收规范 V1

> 文档性质：第一阶段最小质量保障方案  
> 当前目标：优先保证内容能够稳定采集、整理、去重、聚类并生成可读报告，不建设复杂评测平台。

## 1. 设计原则

第一阶段评测只回答三个问题：

1. 同一件事是否被正确整理到一起；
2. 早报、晚报和周报是否只保留真正新增的信息；
3. 最终报告中的重大事实是否有来源、是否清楚、是否有用。

评测不能成为主流程开发的前置阻塞项。只保留少量高价值案例，开发中遇到真实错误后再追加案例。

暂不建设：

- Benchmark Dashboard；
- 自动CI门禁；
- 完整历史数据库Replay；
- 多模型排行榜；
- 独立标注后台；
- 复杂加权总分；
- 大规模人工标注集。

---

## 2. 文件结构

第一阶段只使用：

```text
evaluation/
└─ cases.yaml
```

全部案例统一存放在一个文件中，通过`case_type`区分：

- `clustering`：聚类与去重；
- `novelty`：跨报告增量判断；
- `report`：报告条目、事实支持与表达质量。

案例总量初期控制在30至50个。仓库初始只放一组骨架案例，后续从真实运行错误中逐步补充。

---

## 3. 案例最小结构

```yaml
id: clustering-001
case_type: clustering
description: 同一官方公告被媒体转载
inputs: []
expected:
  relation: duplicate
  cluster_count: 1
reason: 转载没有独立新增事实
```

输入只保存足够判断的结构化摘要，不要求存放完整网页、视频或历史数据库。

案例字段：

```yaml
id: string
case_type: clustering | novelty | report
description: string
inputs: list
expected: object
must_not: list
reason: string
status: active | draft
```

## 4. 初期只检查六项结果

### 4.1 聚类正确率

检查：

- 重复转载是否合并；
- 独立确认是否保留为`corroboration`；
- 同主题不同事件是否错误合并；
- 相反观点是否被错误当作重复。

记录方式：

```text
通过案例数 / 聚类案例总数
```

### 4.2 增量判断正确率

检查：

- `no_change`；
- `minor_detail`；
- `material_update`；
- `reversal`；
- `market_confirmation`。

重点避免无实质更新的内容重复进入晚报。

### 4.3 重大事实支持率

检查核心结论和核心条目中的重大事实是否能够映射到：

- Item；
- 官方数据；
- MarketSnapshot；
- CalendarEvent；
- Evidence Packet中的明确字段。

第一阶段不要求逐句检查全文，只检查重大陈述。

### 4.4 重复问题数

统计：

- 同一Cluster在同一报告不同栏目重复；
- 早报已写且无更新，晚报再次展开；
- 同一观点由多个转载来源重复展示。

### 4.5 重大遗漏数

人工检查是否漏掉：

- 重大官方政策；
- 关键宏观数据；
- 重要方向反转；
- 系统性风险；
- 用户长期重点关注资产的重大事件。

### 4.6 人工综合评分

最终报告采用1至5分人工评分：

| 分数 | 标准 |
|---:|---|
| 5 | 清楚、准确、有明显增量，基本可直接使用 |
| 4 | 整体可用，存在少量表达或排序问题 |
| 3 | 能阅读，但重复、遗漏或解释不足较明显 |
| 2 | 需要大量人工修改 |
| 1 | 事实或结构问题严重，不能使用 |

人工评分只作为辅助，不与其他指标合成为复杂总分。

## 5. 执行方式

第一阶段采用手动或半自动回归：

```text
读取cases.yaml
→ 运行指定模块
→ 保存实际结果
→ 与expected比较
→ 输出通过/失败及原因
```

每次以下变更后建议运行相关案例：

- 聚类Prompt或规则改变；
- Novelty规则改变；
- 报告Prompt改变；
- 数据结构或字段含义改变；
- 更换主要模型；
- 修复重大内容错误。

普通来源新增、文案微调和非核心配置修改不要求全量运行。

---

## 6. 简单结果格式

结果可以暂时输出为Markdown或终端文本：

```markdown
# Evaluation Result

- 聚类：8/10通过
- 增量判断：9/10通过
- 无支持重大事实：0
- 重复问题：1
- 重大遗漏：0
- 人工综合评分：4/5

## 失败案例

- clustering-004：错误合并同主题但不同月份的CPI数据
```

暂不要求存入数据库，也不要求建设可视化页面。

---

## 7. 案例增加规则

只有以下情况才增加新案例：

- 真实运行中出现过错误；
- 该错误可能再次发生；
- 现有案例无法覆盖；
- 能明确写出正确预期。

不要为了增加数量制造大量低价值或高度相似案例。

每次修复生产问题时，优先增加一个能够复现该问题的案例，防止回归。

---

## 8. 第一阶段完成标准

评测模块满足以下条件即可停止扩展并进入主流程实现：

- `cases.yaml`格式稳定；
- 聚类、增量和报告三类均有代表性案例；
- 能人工或脚本化运行案例；
- 能明确列出失败案例；
- 不阻碍采集、整理和报告生成主链路开发。

核心目标始终是先让内容整理系统运行起来。评测用于防止明显退步，而不是追求完整学术基准。
