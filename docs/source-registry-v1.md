## 3. 为什么不一次性启用361个来源

当前仍处于需求和架构设计阶段。一次性启用全部来源会同时放大以下问题：

- 免费X和公众号Provider的失败率与限流风险；
- YouTube字幕与音频转写量；
- 重复内容和AI处理成本；
- 账号停更、改名、失效或方向变化带来的配置污染；
- 尚未验证的低价值与广告内容。

因此采用分阶段启用：

```text
Stage 0  只完成来源治理与配置，不运行采集
Stage 1  验证核心来源的免费Provider
Stage 2  核心来源进入手工与半自动试运行
Stage 3  分批加入supplementary来源
Stage 4  observe来源轮换低频验证
```

登记在Registry中不等于已经启用。实际启用由部署阶段、SourceProviderBinding健康状态和采集验证共同决定。

## 6. 来源角色

来源优先级与来源角色分开维护。角色包括：

- `official_fact`：官方事实；
- `data_release`：原始数据发布；
- `professional_reporting`：专业报道；
- `event_discovery`：事件发现与快讯；
- `research`：机构或深度研究；
- `analysis`：分析与解释；
- `data_analysis`：链上、资金面、量化或结构化数据分析；
- `trading_view`：技术分析、期权和交易观点；
- `education`：方法、知识与工具教学；
- `opinion`：个人观点。

不同角色不能使用同一淘汰标准。例如官方来源即使入选率低也可能必须保留；方法类来源日报入选率低，但可能对周报与研究模块具有价值。

## 7. 尚未完成的验证

当前分级主要依据用户原始标记、定位描述和来源性质，尚未逐一在线核验。正式启用前需要检查：

1. URL、账号和频道ID是否仍有效；
2. 显示名称是否变化；
3. 是否停更、转私密或改变内容方向；
4. 公众号种子文章能否解析真实公众号身份；
5. 跨平台账号是否确属同一创作者；
6. 内容中广告、生活和低价值推广占比；
7. 免费Provider的抓取成功率、时间精度和正文完整率；
8. 原始`***`标记是否仍符合当前关注重点。

`波浪理论学习`在公众号分组中提供的是微博链接，暂不启用，需在决定是否增加微博Collector后再处理。

## 8. 后续优化机制

试运行后按30天与90天窗口记录：

- 总采集数量；
- 可用内容率；
- 独立增量率；
- 重复率；
- 核心事件贡献次数；
- 反方观点贡献次数；
- 完整正文或字幕取得率；
- Provider失败率；
- 事实错误和可靠性问题。

系统只生成升降级建议，不直接自动修改来源级别。建议每两至四周人工确认一次配置调整。

## 9. 当前结论

第一版不追求立即监控全部361个来源，而是：

```text
完整保留来源资产
→ 清理重复和跨平台关系
→ 核心批次先验证
→ 补充来源分批加入
→ 观察来源低频轮换
→ 用真实运行数据持续升降级
```

这样既保留用户长期积累的来源名单，也避免在免费采集条件下让大量未经验证的账号拖累系统稳定性和报告质量。
