# 第二版优化指南

本文件统一收纳所有不属于第一版最小闭环的功能、模型与技术方案。

第一版只负责稳定完成：

```text
信息源管理 → 候选信息收集 → 人工筛选与编辑 → 日报/周报输出 → 历史归档
```

第二版不是第一版的预设开发清单。只有当第一版连续运行后出现明确问题，才从本指南中选择对应能力加入。

## 1. 启动第二版的前提

至少满足以下条件：

- 第一版已连续运行两个完整交易周；
- 核心信息源、日报和周报格式基本稳定；
- 已记录重复操作、时间消耗、漏报和阅读体验问题；
- 能明确说明新增功能要解决的具体问题；
- 新增功能不会显著扩大人工审核负担。

## 2. 优化原则

1. 每次只增加一至两个能力。
2. 先解决高频、机械、耗时的问题。
3. 先优化信息质量，再增加研究复杂度。
4. 新字段必须在真实使用中反复出现后再结构化。
5. AI输出必须保留来源，并接受人工审核。
6. 技术架构随需求升级，不提前建设复杂平台。

## 3. 信息采集与流程自动化

适用问题：每天重复打开固定网站、复制标题和链接、手工归档耗时较多。

可选能力：

- RSS、API和固定网页采集；
- 自动保存标题、发布时间、来源、链接和摘要；
- 邮件与Newsletter导入；
- 抓取失败记录与来源健康检查；
- 基础市场和内容类型分类；
- 标题或链接级重复识别；
- 按模板生成日报草稿；
- 自动归档历史报告；
- 定时任务与运行日志。

建议技术：

- Python脚本；
- GitHub Actions或轻量定时任务；
- JSON、CSV或SQLite；
- Jinja2生成Markdown。

不应在这一阶段直接引入微服务、Kafka或复杂任务编排。

## 4. 内容体系优化

适用问题：日报信息过散、同一主题反复出现、重要数据缺少可比较背景。

可选能力：

- WatchTopic持续跟踪主题；
- 事件状态：新增、更新、纠正、结束；
- 经济数据的前值、预期值和实际值；
- 同一主题的历史进展汇总；
- 个性化市场和主题权重；
- 重点公司、资产和人物观察清单；
- 自动生成市场数据表和关键图表；
- 全文搜索和历史报告检索；
- 多来源交叉验证提示；
- 简要影响分析和后续观察点。

新增字段应优先保持轻量，例如：

```yaml
topic_id:
status: new | updating | closed
previous_context:
new_information:
next_watch:
```

## 5. 去重与事件整理

适用问题：大量媒体重复报道同一事实，人工需要反复阅读和选择。

可分阶段增加：

1. URL和标题精确去重；
2. 文本相似度去重；
3. 同一事件的多来源聚合；
4. 新增信息与旧背景分离；
5. 事件时间线和更新链。

可引入的对象：

- `Document`：原始文章、公告或资料；
- `Event`：一个或多个Document对应的同一事实事件。

只有同事件重复已经成为明显痛点时，才从第一版的`Item`升级为`Document + Event`。

## 6. 研究增强

适用问题：日报和周报已稳定，但需要进一步沉淀研究判断，而不只是整理信息。

可选能力：

- 市场预期与实际结果；
- 预期差描述；
- 资产影响方向和期限；
- 跨资产影响机制；
- 传导链编辑；
- 当前市场叙事；
- 资产和主题观点管理；
- 观点失效条件；
- 后续验证指标；
- 历史观点版本和复盘；
- 判断正确、部分正确、错误或尚未验证的统计。

可能引入的领域对象：

```text
Event
├─ AssetImpact
├─ TransmissionLink
├─ ResearchNote
└─ View
     └─ Evaluation
```

这些对象不应一次性全部加入。优先从自由文本研究备注开始，只有字段高频重复时再结构化。

## 7. 重要性评分升级

第一版只使用`high / medium / low`。

当候选信息数量明显增加、人工排序负担较大时，可以评估0至100分模型。候选维度包括：

- 市场影响；
- 预期差；
- 影响范围；
- 持续时间；
- 来源可靠性；
- 新颖性；
- 可验证性；
- 与个人重点主题的相关性。

评分只能用于排序，不能替代人工判断。必须保留人工覆盖能力，并验证高分信息是否真的更值得进入报告。

## 8. 数据库与搜索

升级顺序建议：

1. Markdown、YAML、JSON或CSV；
2. SQLite：本地搜索、去重和结构化查询；
3. PostgreSQL：多人协作、网页后端或并发任务；
4. 全文检索；
5. 向量检索：仅用于语义搜索、历史相似事件和研究资料召回。

向量数据库不是第二版必选项。普通全文搜索能够解决的问题，不使用向量检索。

## 9. 网页工作台

适用问题：文件和命令行操作已明显降低审核效率，或需要多人协作。

可选页面：

- 今日收集箱；
- 待审核信息队列；
- 原文与摘要对照；
- 来源管理；
- 日报和周报编辑器；
- 持续主题页面；
- 历史搜索；
- 市场数据与图表；
- 研究观点与复盘。

技术可采用FastAPI配合轻量前端，也可以先用Streamlit验证交互。网页化不应改变人工最终审核原则。

## 10. AI能力升级

在第一版摘要、翻译、分类和格式整理基础上，可逐步增加：

- 多来源合并摘要；
- 新增信息识别；
- 事实、评论和推断分离；
- 数据值提取；
- 事件状态判断；
- 主题进展总结；
- 周报归纳；
- 影响机制候选；
- 反向证据和遗漏检查；
- 历史相似事件检索。

多智能体系统仅在单一流程无法清晰维护、任务边界已经稳定时评估。第一版和第二版早期均不以多智能体为目标。

## 11. 第二版候选实施顺序

建议按真实痛点选择，不要求全部实现：

1. 固定来源自动采集；
2. 标题、链接和正文基础去重；
3. 日报草稿和历史归档自动化；
4. 持续主题与事件更新链；
5. 市场数据和关键图表；
6. 全文搜索；
7. 简要影响分析；
8. Event等结构化事件模型；
9. 观点管理与历史验证；
10. 网页工作台、向量检索和更复杂AI流程。

## 12. 单项优化验收模板

每增加一项能力，都应记录：

```markdown
### 优化项

**要解决的问题：**

**当前人工耗时或错误：**

**新增功能：**

**是否增加审核负担：**

**试运行周期：**

**验收指标：**

**最终结论：保留 / 修改 / 删除**
```

第二版的目标不是功能更多，而是在不牺牲可靠性的前提下，提高信息处理效率、报告质量和历史研究价值。
