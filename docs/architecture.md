# 跨资产信息简报系统架构 V1

## 1. 第一版定位

第一版是一套无人值守、定时采集多平台信息并自动生成Markdown报告的系统。

固定交付：

- 每日北京时间08:00生成早报；
- 每日北京时间18:00生成晚报；
- 每周六北京时间18:00生成周报；
- 暂不提供飞书、邮件、微信或网页推送；
- 最终成品保存为Markdown，原始数据和运行状态另外归档。

第一版重点不是建立完整研究平台，而是打通可稳定运行的自动闭环：

```text
持续采集
→ 原始内容归档
→ 标准化
→ 过滤与去重
→ 信息簇合并
→ 自动分类和评分
→ 报告编排与生成
→ 状态更新和日志记录
```

### 1.1 内容与数据成本约束

第一版不购买新闻订阅、付费正文、商业行情、商业宏观日历、X付费API、付费代理或其他内容数据服务。

该约束只针对内容与数据来源，不限制AI模型与Token使用。摘要、翻译、分类、聚类、评分和报告生成可以调用外部AI模型。详细规则见`docs/zero-cost-content-strategy.md`。

## 2. 报告窗口

### 2.1 早报

- 生成时间：北京时间08:00；
- 内容窗口：前一日18:00至当日08:00。

### 2.2 晚报

- 生成时间：北京时间18:00；
- 内容窗口：当日08:00至18:00。

### 2.3 周报

- 生成时间：每周六18:00；
- 内容窗口：上周六18:00至本周六18:00；
- 周报重新综合主题，不拼接日报正文。

内容入选按报告窗口判断；后台采集与报告时间分离，建议每30至60分钟发现和保存一次新内容。

## 3. 总体模块

```text
1. Source Registry      来源注册与分级
2. Collectors           内容、行情、宏观和事件采集器
3. Normalization        统一数据结构与内容获取状态
4. Storage              SQLite状态库与原始文件归档
5. Intelligence         过滤、去重、聚类、分类、摘要和评分
6. Report Engine        早报、晚报和周报编排
7. Scheduler            定时任务、重试、日志和异常隔离
```

第一版使用单一Python项目，不采用微服务、前端工作台、向量数据库或复杂知识图谱。

## 4. Source Registry：来源管理

来源类型包括：

- 官方机构、央行、监管部门和交易所；
- 专业媒体；
- 财经快讯与数据平台；
- X账号；
- 公众号；
- YouTube频道；
- 行情与宏观免费接口或公开页面。

来源运行等级：

- `core`：每个报告周期检查，使用正常入选门槛；
- `supplementary`：正常采集，使用略高门槛；
- `observe`：继续采集，使用明显更高门槛；
- `disabled`：暂不采集，保留配置和停用原因。

来源等级表示可靠性和原始程度，来源优先级表示运行策略；两者都不能保证内容自动进入报告。

每个来源还应记录：

- `provider`：当前使用的具体获取方式；
- `fallback_providers`：免费备用获取方式；
- `content_cost_type`：应固定为`free`或`public`；
- `terms_note`：个人研究用途和来源条款说明；
- `last_success_at`与`failure_count`。

## 5. Collectors：采集器

### 5.1 专业媒体与网页

核心专业来源包括Reuters、Bloomberg、财新、证券时报、S&P Global Commodity Insights、Fastmarkets、CoinDesk、Foresight News和金色财经；金十数据作为财经快讯与宏观日历辅助源。

采集器只使用RSS、公开Feed、公开网页、公开摘要和无需付费的数据接口。付费墙或无法取得正文时，只保存合法可得的标题、摘要和链接，不推断不可访问内容。

媒体名称进入核心来源池，不代表系统一定可以取得全文。报告必须记录实际内容完整度：

- `complete`；
- `partial`；
- `metadata_only`；
- `unavailable`。

### 5.2 X

X官方付费API不作为第一版方案。第一版复用参考项目现有抓取链，并封装多个免费Provider。

内容规则：

- 不抓对其他用户的普通回复；
- 保留作者对自己的连续回复并合并为线程；
- 抓原创、转发和引用；
- 纯转发无新增评论时不单独形成观点；
- 引用帖分别保存原始内容和作者评论；
- 同一线程按顺序合并；
- 使用平台ID、URL、作者时间组合和内容哈希增量去重；
- 时间不精确时标记`approximate`；
- Provider失败时记录覆盖率，不阻塞报告。

推荐Provider结构：

```text
XCollector
├─ ReferenceProjectProvider
├─ NitterCompatibleProvider
├─ PublicSearchProvider
└─ WebPageProvider
```

### 5.3 公众号

- 仅处理发布时间属于报告窗口的新文章；
- 参考项目的公众号发现逻辑作为第一版主Provider；
- 文章发现Provider与微信正文解析器分离；
- 获取正文并提取图片、长图等可用内容；
- 不单独开发付费或广告分类器，而是判断实际取得内容是否具有研究价值；
- 无法取得正文时不根据标题推断观点；
- 免费第三方接口失效时只影响公众号模块。

### 5.4 YouTube

- 使用YouTube Data API免费配额或yt-dlp发现视频；
- 排除直播、直播回放和Shorts；
- 优先获取人工字幕，其次自动字幕；
- 无字幕时使用音频转写；
- 必要时抽取关键画面补充图表和演示信息；
- 音频、字幕和有效画面均不可得时记录失败，不生成猜测性摘要；
- 长视频分片处理，保留时间戳，再生成视频级摘要。

内容数据零付费不限制AI转写模型的Token或计算成本，但不购买第三方字幕库或媒体监控数据。

### 5.5 行情和宏观数据

稳定性优先，并接受免费来源的口径替换、延迟和局部缺失。

第一版推荐：

- 海外权益、ETF、外汇、黄金、白银、原油和COMEX铜：`yfinance`主Provider，公开日线来源作为备用；
- 铜：优先COMEX铜；沪铜通过AkShare或上期所公开页面补充；不强求LME 3个月铜；
- 锂：AkShare或广期所公开页面中的碳酸锂主力；
- BTC、ETH：Binance公开市场数据接口，预留OKX或Bybit公开行情备用；
- 美债2Y、10Y、30Y：美国财政部官方日度数据；
- 中债2Y、10Y、30Y：中债日终收益率曲线；
- 宏观实际值：FRED、BLS、BEA、中国国家统计局、人民银行及其他官方来源；
- 宏观日历：金十公开财经日历与各官方机构发布日程组合；
- 一致预期：只有免费公开页面能够稳定取得时才展示，缺失时不得推断；
- AI板块行情代理：SMH半导体ETF。

系统不追求专业终端级实时行情，市场看板主要使用报告窗口涨跌、固定时点快照、收益率BP变化和可验证的异常波动。

### 5.6 重要事件监控

`docs/event-monitoring-guide.md`记录推荐性判断，`config/event-monitoring.yaml`记录当前运行配置。推荐类别不强制进入报告，清单外重大事件仍可触发例外。

事件监控优先读取官方公开来源，不购买监管、政策或产业事件数据库。

### 5.7 零付费内容来源约束

任何Collector和Provider均不得在免费来源失败时自动切换至付费内容服务。

统一配置：

```yaml
content_data_cost_policy:
  mode: zero_paid_sources
  allow_paid_news: false
  allow_paid_market_data: false
  allow_paid_macro_calendar: false
  allow_paid_social_api: false
  allow_paid_proxy: false
  ai_model_cost_restricted: false
```

## 6. Normalization：统一内容结构

不同来源统一转换为`Item`，至少保留：

- 平台和来源；
- 外部内容ID；
- 标题、正文、原始链接；
- 发布时间、原始时间文本和时间精度；
- 市场、主题和内容性质；
- 内容获取完整性；
- 具体Provider；
- 原始文件位置；
- 处理状态和错误信息。

时间精度区分：

- `exact`：可获得准确时间；
- `approximate`：只有相对时间或估算时间；
- `unknown`：无法可靠确定。

内容完整度区分：

- `complete`：完整正文、字幕或数据；
- `partial`：只取得部分正文或字段；
- `metadata_only`：只有标题、时间和链接；
- `unavailable`：本次获取失败。

## 7. Storage：存储方案

第一版使用：

```text
SQLite：来源、状态、去重、信息簇、报告引用和运行记录
JSON/Text：原始抓取结果、正文、字幕和调试材料
Markdown：早报、晚报和周报成品
```

建议目录：

```text
news-stugy/
├─ config/
├─ docs/
├─ reference/
├─ src/
│  ├─ collectors/
│  ├─ providers/
│  ├─ normalization/
│  ├─ intelligence/
│  ├─ reports/
│  ├─ storage/
│  └─ scheduler/
├─ data/
│  ├─ app.db
│  ├─ raw/YYYY-MM-DD/
│  ├─ processed/YYYY-MM-DD/
│  └─ logs/
├─ reports/
│  ├─ daily/
│  │  ├─ morning/
│  │  └─ evening/
│  └─ weekly/
└─ tests/
```

参考项目放入`reference/`并作为采集与预处理原型。正式代码可迁移通用逻辑和测试思路，但不继续在实验脚本上叠加全部功能。

## 8. Intelligence：信息处理

### 8.1 硬过滤

先排除不属于窗口、X普通回复、直播、Shorts、完全重复、纯生活和无研究价值推广等内容。

`metadata_only`内容通常只能作为事件线索，不能生成详细观点摘要。

### 8.2 确定性去重

优先级：

1. 平台、作者和外部ID；
2. 标准化URL；
3. 正文哈希；
4. 作者、时间和标题；
5. 标准化内容哈希。

### 8.3 信息簇

报告不直接处理每条Item，而是处理`Cluster`：

```text
Source → Item → Cluster → Report
```

信息簇关系包括：

- `duplicate`；
- `corroboration`；
- `new_detail`；
- `update`；
- `market_reaction`；
- `industry_impact`；
- `commentary`；
- `contradiction`。

同一事件的新增细节应合并，相反观点应并列，不得平均化。系统同时记录总来源数量和独立来源数量。

### 8.4 自动分类和评分

评分对象为Cluster，日报和周报使用不同权重。总分只用于候选分层，重大事件检查、事实可靠性和报告篇幅仍需单独判断。详细规则见`config/scoring.yaml`。

免费来源的抓取完整度不能直接等同于事件重要性，但证据不完整会限制报告可以作出的事实陈述。

### 8.5 跨报告更新判断

已进入早报的事件，晚报只有在出现`material_update`、`reversal`或`market_confirmation`时再次报告。周报按一周内事件演变重新综合。

## 9. Report Engine：报告生成

报告生成器读取：

- 已筛选的信息簇；
- 跨资产市场快照；
- 已发生宏观数据和事件；
- 后续宏观与政策日历；
- 来源完整性和运行异常。

报告只展示能被原始记录支持的事实。观点、解释和预测必须明确区分；无法确认驱动时使用谨慎表述。

免费来源未能提供预期值、完整正文或准确时间时，应省略对应判断或明确标记缺失，不为维持模板完整而补写。

具体结构见`docs/report-spec.md`。

## 10. Scheduler：运行与异常

### 10.1 调度

- 内容发现：每30至60分钟；
- 早报生成：每日08:00；
- 晚报生成：每日18:00；
- 周报生成：每周六18:00；
- 时区统一使用`Asia/Shanghai`。

不同来源可以使用不同频率：核心快讯和X更高频，公众号和YouTube较低频，官方日终数据按发布时间刷新。

### 10.2 异常隔离

- 单个来源失败不得阻塞其他来源；
- 采集器按来源重试并设置超时；
- 同类免费Provider可以自动回退；
- 禁止自动回退到付费内容Provider；
- AI调用失败时允许分批重试；
- 部分来源失败仍应按时生成报告；
- 不支持的事实应失败关闭，不得自动补写；
- 报告可附数据完整性说明。

## 11. 第一版不包含

- 飞书、邮件或微信推送；
- 网页前端和用户系统；
- 付费内容和商业数据源；
- 博主全历史深度画像；
- 向量数据库和复杂语义搜索；
- 长期知识图谱和自动因果网络；
- 自动交易建议或仓位建议；
- 完整实时行情终端。

## 12. 第一版验收标准

连续运行两个完整交易周并达到：

- 早报、晚报和周报按时生成；
- 四类内容来源和行情、宏观数据均形成可替换采集器；
- 不产生任何内容与数据订阅费用；
- 原始内容、正文、字幕和链接可追溯；
- 同一事件在单份及跨份报告中不无意义重复；
- 事实、分析、观点和预测能够明确区分；
- 单个来源失败不影响整份报告；
- 报告篇幅和信息密度符合规范；
- 记录来源成功率、完整正文取得率、重复率、低价值率和漏报案例；
- 记录行情缺失率与宏观预期值覆盖率；
- 试运行后至少完成一次来源、评分和事件监控配置复盘。
