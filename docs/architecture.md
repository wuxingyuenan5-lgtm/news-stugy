# 跨资产信息简报系统架构 V1

## 1. 第一版定位

第一版是一套无人值守、定时采集多平台信息并自动生成Markdown报告的系统。

固定交付：

- 每日北京时间08:00生成早报；
- 每日北京时间18:00生成晚报；
- 每周六北京时间18:00生成周报；
- 暂不提供飞书、邮件、微信或网页推送；
- 最终成品保存为Markdown，原始数据和运行状态另外归档。

第一版重点不是建立完整研究平台，而是打通可稳定运行的自动闭环：

```text
持续采集
→ 原始内容归档
→ 标准化
→ 过滤与去重
→ 信息簇合并
→ 自动分类和评分
→ 报告编排与生成
→ 状态更新和日志记录
```

## 2. 报告窗口

### 2.1 早报

- 生成时间：北京时间08:00；
- 内容窗口：前一日18:00至当日08:00。

### 2.2 晚报

- 生成时间：北京时间18:00；
- 内容窗口：当日08:00至18:00。

### 2.3 周报

- 生成时间：每周六18:00；
- 内容窗口：上周六18:00至本周六18:00；
- 周报重新综合主题，不拼接日报正文。

内容入选按报告窗口判断；后台采集与报告时间分离，建议每30至60分钟发现和保存一次新内容。

## 3. 总体模块

```text
1. Source Registry      来源注册与分级
2. Collectors           内容、行情、宏观和事件采集器
3. Normalization        统一数据结构与内容获取状态
4. Storage              SQLite状态库与原始文件归档
5. Intelligence         过滤、去重、聚类、分类、摘要和评分
6. Report Engine        早报、晚报和周报编排
7. Scheduler            定时任务、重试、日志和异常隔离
```

第一版使用单一Python项目，不采用微服务、前端工作台、向量数据库或复杂知识图谱。

## 4. Source Registry：来源管理

来源类型包括：

- 官方机构、央行、监管部门和交易所；
- 专业媒体；
- 财经快讯与数据平台；
- X账号；
- 公众号；
- YouTube频道；
- 行情与宏观API。

来源运行等级：

- `core`：每个报告周期检查，使用正常入选门槛；
- `supplementary`：正常采集，使用略高门槛；
- `observe`：继续采集，使用明显更高门槛；
- `disabled`：暂不采集，保留配置和停用原因。

来源等级表示可靠性和原始程度，来源优先级表示运行策略；两者都不能保证内容自动进入报告。

## 5. Collectors：采集器

### 5.1 专业媒体与网页

核心专业来源包括Reuters、Bloomberg、财新、证券时报、S&P Global Commodity Insights、Fastmarkets、CoinDesk、Foresight News和金色财经；金十数据作为财经快讯与宏观日历辅助源。

采集器优先使用RSS、公开API和稳定网页接口。付费墙或无法取得正文时，只保存合法可得的标题、摘要和链接，不推断不可访问内容。

### 5.2 X

- 不抓回复；
- 抓原创、转发和引用；
- 纯转发无新增评论时不单独形成观点；
- 引用帖分别保存原始内容和作者评论；
- 同一线程按顺序合并；
- 使用平台ID、URL和内容哈希增量去重。

### 5.3 公众号

- 仅处理发布时间属于报告窗口的新文章；
- 获取正文并提取图片、长图等可用内容；
- 不单独开发付费或广告分类器，而是判断实际取得内容是否具有研究价值；
- 无法取得正文时不根据标题推断观点。

### 5.4 YouTube

- 排除直播和Shorts；
- 优先获取官方字幕，其次自动字幕；
- 无字幕时使用音频转写；
- 必要时抽取关键画面补充图表和演示信息；
- 音频、字幕和有效画面均不可得时记录失败，不生成猜测性摘要。

### 5.5 行情和宏观数据

稳定性优先，第一版允许按供应商能力替换具体标的口径：

- 海外权益、ETF、外汇、黄金、白银、原油：统一成熟行情API；
- 铜：优先COMEX铜或沪铜主力，不强求LME 3个月铜；
- 锂：广期所碳酸锂主力；
- BTC、ETH：Binance公开API；
- 美债2Y、10Y、30Y：美国财政部官方日度数据；
- 中债2Y、10Y、30Y：中债日终收益率曲线；
- 宏观实际值：FRED、BLS、BEA及其他官方来源；
- 宏观日历和预期值：成熟经济日历API；
- AI板块行情代理：SMH半导体ETF。

### 5.6 重要事件监控

`docs/event-monitoring-guide.md`记录推荐性判断，`config/event-monitoring.yaml`记录当前运行配置。推荐类别不强制进入报告，清单外重大事件仍可触发例外。

## 6. Normalization：统一内容结构

不同来源统一转换为`Item`，至少保留：

- 平台和来源；
- 外部内容ID；
- 标题、正文、原始链接；
- 发布时间、原始时间文本和时间精度；
- 市场、主题和内容性质；
- 内容获取完整性；
- 原始文件位置；
- 处理状态和错误信息。

时间精度区分：

- `exact`：可获得准确时间；
- `approximate`：只有相对时间或估算时间；
- `unknown`：无法可靠确定。

## 7. Storage：存储方案

第一版使用：

```text
SQLite：来源、状态、去重、信息簇、报告引用和运行记录
JSON/Text：原始抓取结果、正文、字幕和调试材料
Markdown：早报、晚报和周报成品
```

建议目录：

├─ docs/

## 8. Intelligence：信息处理

### 8.1 硬过滤

先排除不属于窗口、X回复、直播、Shorts、完全重复、纯生活和无研究价值推广等内容。

### 8.2 确定性去重

优先级：

1. 平台、作者和外部ID；
2. 标准化URL；
3. 正文哈希；
4. 作者、时间和标题；
5. 标准化内容哈希。

### 8.3 信息簇

报告不直接处理每条Item，而是处理`Cluster`：

```text
Source → Item → Cluster → Report
```

信息簇关系包括：

- `duplicate`；
- `corroboration`；
- `new_detail`；
- `update`；
- `market_reaction`；
- `industry_impact`；
- `commentary`；
- `contradiction`。

同一事件的新增细节应合并，相反观点应并列，不得平均化。系统同时记录总来源数量和独立来源数量。

### 8.4 自动分类和评分

评分对象为Cluster，日报和周报使用不同权重。总分只用于候选分层，重大事件检查、事实可靠性和报告篇幅仍需单独判断。详细规则见`config/scoring.yaml`。

### 8.5 跨报告更新判断

已进入早报的事件，晚报只有在出现`material_update`、`reversal`或`market_confirmation`时再次报告。周报按一周内事件演变重新综合。

## 9. Report Engine：报告生成

报告生成器读取：

- 已筛选的信息簇；
- 跨资产市场快照；
- 已发生宏观数据和事件；
- 后续宏观与政策日历；
- 来源完整性和运行异常。

报告只展示能被原始记录支持的事实。观点、解释和预测必须明确区分；无法确认驱动时使用谨慎表述。

具体结构见`docs/report-spec.md`。

## 10. Scheduler：运行与异常

### 10.1 调度

- 内容发现：每30至60分钟；
- 早报生成：每日08:00；
- 晚报生成：每日18:00；
- 周报生成：每周六18:00；
- 时区统一使用`Asia/Shanghai`。

### 10.2 异常隔离

- 单个来源失败不得阻塞其他来源；
- 采集器按来源重试并设置超时；
- AI调用失败时允许分批重试；
- 部分来源失败仍应按时生成报告；
- 不支持的事实应失败关闭，不得自动补写；
- 报告可附数据完整性说明。

## 11. 第一版不包含

- 飞书、邮件或微信推送；
- 网页前端和用户系统；
- 博主全历史深度画像；
- 向量数据库和复杂语义搜索；
- 长期知识图谱和自动因果网络；
- 自动交易建议或仓位建议；
- 完整实时行情终端。

## 12. 第一版验收标准

连续运行两个完整交易周并达到：

- 早报、晚报和周报按时生成；
- 四类内容来源和行情、宏观数据均形成可替换采集器；
- 原始内容、正文、字幕和链接可追溯；
- 同一事件在单份及跨份报告中不无意义重复；
- 事实、分析、观点和预测能够明确区分；
- 单个来源失败不影响整份报告；
- 报告篇幅和信息密度符合规范；
- 记录来源成功率、重复率、低价值率和漏报案例；
- 试运行后至少完成一次来源、评分和事件监控配置复盘。
