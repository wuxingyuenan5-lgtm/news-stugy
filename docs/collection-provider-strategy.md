# 内容与数据采集及 Provider 策略 V1

> 文档性质：采集层设计规范，不代表已经开始代码实现  
> 当前版本：0.1  
> 当前阶段：需求与架构设计  
> 适用范围：专业媒体、X、微信公众号、YouTube、行情、宏观数据、宏观日历与政策事件

## 1. 文档目的

本文件用于统一第一版系统的采集技术路线、参考项目复用边界、免费 Provider 选择顺序和降级规则。

当前阶段只完善设计，不急于落地采集代码。后续正式开发前，应先依据本文件验证现有参考项目和成熟免费工具的实际可用性，再决定需要自行开发的部分。

核心原则：

```text
成熟免费项目或官方免费接口
→ 已验证的参考项目代码
→ 通用免费网页解析工具
→ 项目内轻量适配器
→ 最后才考虑自行实现底层抓取
```

第一版不购买内容和数据服务，但AI模型、Token和AI转写成本不受该约束。详细成本边界见`docs/zero-cost-content-strategy.md`与`config/content-cost-policy.yaml`。

## 2. 总体决策原则

### 2.1 成熟免费项目优先

已经存在成熟免费工具时，不重复开发其底层能力。正式项目主要负责：

- 配置目标来源；
- 调用和包装第三方工具；
- 转换统一数据结构；
- 保存游标、增量状态和原始记录；
- 记录失败、限流和内容完整度；
- 在同类免费Provider之间切换。

### 2.2 参考项目优先实测，再决定重构

当前仓库中的参考项目已具备部分可运行能力。不能仅因其目录、路径或结构不规范，就假设底层方案不可用。

正确流程为：

```text
识别现有能力
→ 找回缺失依赖
→ 使用真实目标列表实测
→ 记录成功率和缺陷
→ 稳定部分直接封装
→ 只重构真实存在的问题
```

### 2.3 Collector、Provider与业务处理解耦

三层职责必须分开：

```text
Source Registry：决定抓取哪些来源
Collector：协调抓取窗口、增量状态和失败处理
Provider：负责某一种具体获取方式
Intelligence：决定内容是否有价值及如何进入报告
```

Provider不得承担重要性评分、跨来源聚类和报告写作职责。

### 2.4 免费来源失败时允许缺失

当免费来源不可用时：

- 可以切换其他免费Provider；
- 可以将内容标记为`partial`、`metadata_only`或`unavailable`；
- 可以在报告中说明完整度不足；
- 不得自动切换付费内容或数据服务；
- 不得依据标题或残缺字段补写事实。

### 2.5 当前阶段不开始大规模编码

在以下事项完成前，不进入大规模采集代码开发：

1. 现有参考项目的真实运行验证；
2. X底层组件是否能够找回；
3. 公众号第三方接口是否仍可使用；
4. 各类免费行情源的口径与稳定性对比；
5. 采集完整度与失败处理标准最终确认；
6. 部署环境和运行方式确定。

## 3. 统一采集结构

概念结构：

```text
Source
  ↓
Collector
  ↓
Primary Provider
  ↓ 失败
Fallback Provider 1
  ↓ 失败
Fallback Provider 2
  ↓
RawItem
  ↓
Normalization
```

单个Provider输出至少应包含：

```yaml
provider_name:
platform:
source_id:
external_item_id:
author:
title:
raw_text:
raw_html:
published_at:
published_at_raw:
time_precision: exact | approximate | unknown
url:
referenced_items: []
content_status: complete | partial | metadata_only | unavailable
collected_at:
raw_payload_path:
error:
metadata: {}
```

Provider输出的是原始采集结果，不是最终报告条目。

## 4. 参考项目复用判断

### 4.1 可直接迁移或改造的能力

- 时间窗口构建；
- 多种发布时间格式解析；
- 按报告窗口过滤；
- 多账号批量运行；
- `seen_keys`增量去重思路；
- 目标账号与脚本分离；
- 原始JSON归档；
- Markdown调试输出；
- 公众号种子文章初始化；
- 公众号`biz`和身份信息解析；
- 公众号指定日期范围文章查询；
- 单条失败继续处理的机制；
- 基础测试和调试思路。

### 4.2 需要重新封装的部分

- Windows绝对路径；
- 环境变量与本地依赖；
- X底层`nitter_client`接入；
- 公众号第三方接口调用；
- JSON状态迁移至SQLite；
- Collector与Provider接口；
- 统一日志、超时、重试和失败隔离；
- 原始数据目录和文件命名；
- 内容完整度与时间精度字段。

### 4.3 不继续作为正式逻辑的部分

- 只能输出单一市场分类的关键词逻辑；
- 根据文本长度和标点形成的简单`signal_score`；
- 仅凭文本是否以`@`开头判断X回复；
- 将采集、分类、摘要和渲染写在同一流程；
- 某一个第三方接口失败便退出全部任务；
- 依赖搜索摘要生成完整观点。

## 5. X采集策略

### 5.1 当前判断

参考项目的X上层流程已经相对完整，包括：

- 账号配置；
- 多账号批处理；
- 时间解析与窗口过滤；
- 增量状态；
- JSON和Markdown输出；
- Nitter链路失败后的搜索回退。

当前主要缺口是负责实际时间线抓取的`nitter_client.py`没有完整收进仓库，而是从本地Windows路径导入。

因此，X第一项工作不是重写爬虫，而是寻找并恢复原有底层组件。

### 5.2 推荐Provider顺序

```text
ReferenceProjectProvider
→ NitterCompatibleProvider
→ PublicSearchProvider
→ WebPageProvider
```

说明：

- `ReferenceProjectProvider`：恢复原项目真实使用的时间线抓取能力；
- `NitterCompatibleProvider`：采用成熟免费兼容项目或实例；
- `PublicSearchProvider`：通过公开搜索索引发现新帖子；
- `WebPageProvider`：在可行时读取公开页面或单帖内容。

### 5.3 必须补强的结构化字段

- 帖子类型：原创、引用、转发、回复；
- 原始帖子ID和链接；
- 被引用或被转发内容；
- 回复目标作者；
- 线程标识与顺序；
- 精确或近似发布时间；
- 正文完整度；
- 媒体附件元数据。

### 5.4 内容规则

- 排除回复其他用户的普通互动；
- 保留作者回复自己的连续线程；
- 原创帖正常处理；
- 引用帖分别保存原内容和作者新增评论；
- 纯转发可以作为发现、热度和佐证信号，但通常不形成独立观点；
- 搜索片段只能作为线索或部分内容，不能视为完整帖子；
- 无法确认时间的帖子不应强行纳入严格报告窗口。

### 5.5 正式编码前验证指标

- 每个账号是否能获取最近帖子；
- 帖子ID和URL完整率；
- 发布时间准确率；
- 原创、引用、转发和回复识别率；
- 自回复线程完整率；
- 报告窗口漏抓率；
- Provider单账号失败率；
- 同一内容重复率。

## 6. 微信公众号采集策略

### 6.1 当前判断

参考项目中的`wx_public_monitor.py`已经形成较完整的公众号发现流程：

```text
种子文章
→ 解析公众号身份与biz
→ 查询指定日期范围的文章URL
→ 重新解析每篇文章
→ 校验公众号归属
→ 按真实发布时间排序
```

因此公众号第一版不应重新设计底层文章发现方式，而应将现有逻辑封装为主Provider。

### 6.2 推荐结构

```text
WeChatCollector
├─ ReferenceWxPublicProvider
├─ BackupDiscoveryProvider
└─ WeChatArticleParser
```

文章发现和正文解析必须分开。发现Provider只负责返回文章URL和基本元数据；ArticleParser负责读取文章正文、图片和精确发布时间。

### 6.3 可直接复用

- 种子文章解析；
- `biz`、昵称和账号身份提取；
- 指定日期范围查询；
- URL去重；
- 公众号归属校验；
- 真实发布时间重新排序；
- 单篇解析失败继续运行；
- JSON输出。

### 6.4 需要补强

- 正文HTML和纯文本清洗；
- 正文图片及长图记录；
- 图片内容是否已处理的状态；
- 尾部广告、推荐阅读和菜单噪音清理；
- SQLite增量状态；
- 第三方接口健康检查；
- 连续失败后的自动降频；
- 文章URL、标题和发布时间的多字段去重。

### 6.5 正式编码前验证指标

- 已有APP ID与安全密钥是否可用；
- 目标公众号识别成功率；
- 近期文章发现完整率；
- 真实发布时间准确率；
- 正文成功取得率；
- 图片和长图可访问率；
- 第三方接口平均延迟与失败率；
- 不同公众号之间的误匹配率。

## 7. YouTube采集策略

### 7.1 当前判断

参考项目中只有YouTube历史样例数据，没有形成自动抓取闭环，因此需要新建正式采集器。

但不自行开发视频下载、字幕下载和音频处理底层能力，而采用成熟免费工具组合。

### 7.2 视频发现

推荐顺序：

```text
YouTube Data API免费配额
→ yt-dlp频道列表
```

负责取得：

- 视频ID；
- 标题；
- 发布时间；
- 时长；
- 直播状态；
- 频道信息；
- 原始链接。

### 7.3 内容取得

推荐顺序：

```text
youtube-transcript-api
→ yt-dlp人工字幕
→ yt-dlp自动字幕
→ 下载音频后AI转写
```

AI转写可以调用外部模型，不受内容数据零付费约束。

### 7.4 排除规则

- 排除正在直播的视频；
- 排除直播回放；
- 排除Shorts；
- 仅处理发布时间属于报告窗口的视频；
- 无字幕、无音频且无有效内容时，不根据标题猜测摘要。

### 7.5 长视频处理

```text
字幕按时间切片
→ 分片提取事实、观点、数据和预测
→ 视频内部去重
→ 合并为视频级摘要
→ 参与跨来源聚类
```

重要内容应尽量保留视频时间戳。

### 7.6 正式编码前验证指标

- 新视频发现率；
- 发布时间准确率；
- 直播与Shorts识别率；
- 人工字幕获取率；
- 自动字幕获取率；
- 音频下载与转写成功率；
- 长视频处理时间；
- 中文、英文及混合语言转写质量。

## 8. 专业媒体与普通网页策略

### 8.1 推荐获取顺序

```text
官方RSS或公开Feed
→ 官网公开栏目和文章页
→ 通用正文提取器
→ 搜索发现与公开摘要
```

### 8.2 通用能力优先

优先建立少量通用Provider：

- RSS/Atom Provider；
- Sitemap或列表页Provider；
- 普通网页Provider；
- 正文提取Provider；
- 搜索发现Provider。

只有在核心来源持续无法取得且有明显价值时，才为单一网站开发特殊适配器。

### 8.3 付费墙处理

- 不绕过付费墙；
- 公开正文可正常摘要；
- 部分公开内容标记`partial`；
- 只有标题、摘要和链接时标记`metadata_only`；
- 不根据标题还原不可访问正文；
- 核心事实尽量回溯官方文件或其他可访问的一手来源。

### 8.4 媒体进入来源池不等于保证全文

Reuters、Bloomberg、财新等可以作为事件发现和来源确认对象，但每条内容能否用于详细摘要，取决于实际取得的内容完整度。

## 9. 行情与宏观数据策略

### 9.1 总原则

- 官方免费数据优先；
- 成熟免费项目优先；
- 报告用途优先于专业终端用途；
- 接受延时、日频和少量缺失；
- 不为填满表格引入来源不明数据。

### 9.2 海外市场

推荐：

- 海外指数、ETF、外汇、黄金、白银、原油、COMEX铜：`yfinance`或其他成熟免费行情项目；
- 同类公开日线数据作为备用；
- 只用于报告窗口变化和固定时点快照，不定位为实时下单数据。

### 9.3 中国市场与商品期货

推荐：

- AkShare等成熟免费项目作为聚合入口；
- 上期所、广期所及其他交易所公开页面或下载文件作为官方补充；
- 沪铜主力和碳酸锂主力按稳定性选择具体口径；
- 不强求难以稳定免费取得的LME 3个月铜。

### 9.4 加密行情

推荐：

```text
Binance公开市场数据
→ OKX公开行情
→ Bybit公开行情
```

至少覆盖BTC、ETH价格、K线、成交量和报告窗口涨跌。资金费率和持仓量只在稳定取得时纳入。

### 9.5 国债收益率

- 美债2Y、10Y、30Y：美国财政部官方日度数据；
- 可选美债10年实际利率；
- 中债2Y、10Y、30Y：中债公开日终收益率曲线；
- 报告明确区分日终数据与盘中数据。

### 9.6 宏观实际值

优先：

- FRED；
- BLS；
- BEA；
- 美联储与美国财政部；
- 中国国家统计局；
- 中国人民银行；
- 海关总署、财政部及其他官方来源。

### 9.7 宏观日历

采用：

```text
金十公开财经日历
+ 央行、统计机构、财政部门和交易所官方日程
```

不购买商业宏观日历和一致预期。

免费渠道能稳定取得一致预期时可以展示；无法取得时只展示前值、实际值、事件时间和市场反应，并标记预期值缺失。

## 10. 重要政策与事件采集

政策、监管和产业事件优先读取：

1. 政府、央行和监管机构；
2. 交易所和公司公告；
3. 法律、规则、咨询文件和正式日历；
4. 专业媒体；
5. 垂直媒体与研究者；
6. 社交媒体线索。

事件类别和推荐优先级见：

- `docs/event-monitoring-guide.md`；
- `config/event-monitoring.yaml`。

事件监控是推荐性的，不代表某一类别每次出现都必须进入报告。

## 13. 正式开发前验证阶段

### 阶段A：参考项目恢复

- 寻找`nitter_client.py`及相关依赖；
- 确认公众号APP ID和安全密钥；
- 使用真实账号名单分别运行X与公众号；
- 保存未经AI加工的原始结果。

### 阶段B：免费工具可行性验证

- 测试YouTube频道发现；
- 测试字幕、自动字幕与音频转写；
- 测试yfinance和AkShare覆盖标的；
- 测试中债和交易所公开页面；
- 测试金十日历和官方日程组合。

### 阶段C：形成验证报告

每个Provider记录：

- 成功率；
- 完整率；
- 时间准确率；
- 链接可回溯率；
- 平均延迟；
- 限流和封锁情况；
- 依赖和部署要求；
- 适合作为主Provider还是备用Provider。

完成验证后再冻结第一版采集技术选型。

## 14. 第一版采集验收标准

- 内容和数据来源不产生订阅费用；
- 核心来源均有明确主Provider；
- 高风险来源至少预留一个免费备用Provider；
- 参考项目稳定能力得到直接复用；
- 单个Provider失败不阻塞其他来源和报告；
- 所有原始内容能够追溯Provider、时间和链接；
- 内容完整度和时间精度有明确标记；
- 免费来源缺失时不编造、不偷换口径；
- 报告能够记录主要来源失败和数据缺失；
- 试运行后可以根据统计结果升级、降级或停用Provider。

## 15. 当前设计结论

第一版正式采用：

```text
成熟免费项目作为底层能力
+ 参考项目作为第一批已验证原型
+ 自研统一数据、智能处理和报告系统
```

具体结论：

- X：先找回和验证参考项目底层抓取组件，不立即重写；
- 公众号：参考项目作为第一版主Provider，重点补正文和状态管理；
- YouTube：使用成熟免费工具组合开发新采集器；
- 专业媒体：通用RSS和网页Provider优先；
- 行情与宏观：成熟免费项目与官方公开数据优先；
- 自研重点放在统一结构、去重、聚类、评分、状态和报告生成；
- 当前继续进行设计优化，不立即进入大规模代码实现。
